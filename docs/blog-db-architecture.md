# 건강 저널 — 데이터 저장 아키텍처 (v0.2)

이수한의원 건강 저널 CMS의 데이터를 "무엇을 / 어디에 / 어떻게" 저장할지 트레이드오프와
확장성까지 따져 확정하는 문서. v0.1(하이브리드 스케치)을 의사결정 가능한 수준으로 끌어올림.

- 계정 올투원 `018789813499` · 리전 `ap-northeast-2` · 버킷 `isuhani-clinic-data`
- 현재 글 1,042개. 증가 속도 느림(병원 블로그, 주 수 편)
- 편집자 = 비개발자 직원 다수(의사 + 조무사). **동시 편집 발생**
- 인증 = 관리자 공유 비번 1개(현행)

---

## 0. 먼저 — 이 시스템의 진짜 제약

설계를 가르는 핵심 사실 세 가지. 일반 "DB 고르기"와 다르다.

1. **공개 사이트는 정적이다.** 발행하면 CodeBuild가 S3 `posts/`를 받아 Next.js로 정적
   export → CloudFront 서빙. 즉 **독자 요청은 DB를 안 친다.** 읽기 트래픽이 100배 늘어도
   DB 부하는 0. → "읽기 확장성"은 이 시스템의 고민거리가 아니다.
2. **DB의 실사용자는 직원 몇 명.** 저장소는 "수백만 독자를 먹이는 read DB"가 아니라
   "직원 5명이 글을 쓰고 고치는 editorial DB"다. → 최적화 축이 **읽기 성능이 아니라
   편집 안전성·정확성·확장 가능성**으로 바뀐다.
3. **본문은 무겁고, 메타는 가볍다.** 본문 HTML은 사진 링크까지 합쳐 글당 수십~수백 KB.
   메타(제목·날짜·카테고리·상태)는 글당 ~1KB. 둘의 접근 패턴이 완전히 다르다(목록은
   메타만, 상세는 본문까지).

> 결론 선취: 비용·읽기성능은 어느 안이든 사실상 무시 가능. 결정은 **동시편집 안전성 +
> 부분수정 + 확장성**에서 갈린다.

## 1. 데이터 모델 (엔티티 · 크기 · 접근 패턴)

| 엔티티 | 크기 | 쓰기 빈도 | 읽기 패턴 |
|---|---|---|---|
| 글 메타 | ~1KB | 낮음(편집 시) | 목록·필터(자주), 단건 |
| 글 본문 HTML | 수십~수백 KB | 낮음 | 상세 열 때만 1건 |
| 이미지 | 수 KB~수 MB | 업로드 시 | CDN 서빙(독자) |
| 카테고리 | 작은 목록 1개 | 드묾 | 전체 로드 |

핵심 접근 패턴(쿼리):
- Q1 최신순 목록 (페이지네이션)
- Q2 카테고리별 목록
- Q3 상태별(발행/임시) 필터 — 직원 관리용
- Q4 제목 검색 — 현재는 클라이언트 필터(소량이라 OK)
- Q5 글 1개 조회(메타+본문)
- Q6 글 추가·수정(부분)·삭제(소프트)

## 2. 후보 3안

### A안 — S3 단독 (현행: `index.json` + `posts/{id}.json`)
- 메타 카탈로그 = 단일 `index.json`, 본문 = 글별 파일.
- 글 추가/수정 = 본문 파일 쓰기 + `index.json` 통째 재작성.

### B안 — 하이브리드: DynamoDB(메타) + S3(본문+이미지)  ← v0.1 방향
- 메타 = DynamoDB 아이템(글=레코드), 본문 = S3 `posts/{id}.json`, 이미지 = S3.
- 목록/필터 = DynamoDB 쿼리, 상세 = DynamoDB + S3 본문.

### C안 — 관계형(RDS/Aurora Serverless, SQLite)
- 정규화 테이블. SQL 쿼리·조인·풀텍스트.

## 3. 트레이드오프 매트릭스

| 축 | A: S3 단독 | B: 하이브리드 DDB+S3 | C: 관계형 |
|---|---|---|---|
| 비용(이 규모) | 최저(거의 0) | 거의 0(월 푼돈) | 최저라도 인스턴스/Aurora 상시비용 ↑ |
| 본문 DB저장 낭비 | 없음(S3) | 없음(S3) | 본문 넣으면 부담, 빼면 하이브리드와 동일 |
| **동시편집 안전** | ✗ 약함(공유 index.json 덮어쓰기) | ✓ 강함(글=독립 레코드, 조건부 update) | ✓ 강함(트랜잭션) |
| **필드 부분수정** | ✗ 파일 통째 재작성 | ✓ `UpdateItem` 필드만 원자적 | ✓ `UPDATE ... SET col` |
| 상태/카테고리 필터 | △ index 전체 로드 후 필터 | ✓ GSI 쿼리 | ✓ WHERE/인덱스 |
| 제목 검색 | △ 클라이언트 필터 | △ 클라이언트/스캔(소량 OK) | ✓ LIKE/FTS |
| 운영 복잡도 | ✓ 최소(서비스 0개 추가) | △ 테이블+GSI 관리 | ✗ 서버/패치/백업 관리 |
| 서버리스 적합 | ✓ | ✓ | ✗ (상시 인스턴스 = 선호와 충돌) |
| 마이그레이션 비용 | 0(현행) | 소(스크립트 거의 완성) | 대 |
| 확장성(아래 §4) | △ | ✓ | ✓(but 과함) |

읽기 성능 축은 **일부러 뺐다** — §0대로 공개 읽기는 정적이라 무의미.

C안은 여기서 탈락: 상시 비용 + 서버 운영이 영광님 서버리스 선호와 정면 충돌하고, 이 규모에
관계형의 조인/트랜잭션 이점이 필요 없다. 풀텍스트 검색이 정말 필요해지면 그건 RDS가 아니라
OpenSearch를 옆에 붙이는 게 정석(별도 §4).

→ **실질 결정은 A vs B.**

## 4. 확장성 시나리오 — 미래에 뭐가 깨지나

| 시나리오 | A: S3 단독 | B: 하이브리드 |
|---|---|---|
| 글 1k→10k | index.json 비대 → 수정마다 통째 재작성 무거워짐, 충돌 위험 ↑ | GSI 쿼리는 O(페이지) 그대로. 무난 |
| 직원 동시 편집 ↑ | 덮어쓰기 사고 증가(치명) | 레코드 단위라 영향 없음 |
| 발행/임시/예약발행 상태 | index에 필드 추가하나 필터가 전부 풀스캔 | `status` GSI로 깔끔. 예약발행도 `publishAt` 속성+조회로 확장 |
| 작성자/감사로그(누가 언제 수정) | 파일엔 못 남김(별도 설계 필요) | 레코드에 `author`,`updatedBy` 필드·이력 아이템으로 자연 확장 |
| 콘텐츠 타입 추가(공지·FAQ·의료진·진료안내) | 프리픽스로 폴더 나눔(관리 흩어짐) | 단일 테이블 PK 패턴으로 흡수 가능(§5 주) |
| 제목·본문 검색 | 한계 → 결국 외부 색인 필요 | 동일 → OpenSearch를 **읽기 전용 색인**으로 옆에 부착(둘 다 같은 해법) |
| 공개 즉시반영(재빌드 제거) | 어려움(정적 전제) | 가능(동적 읽기 전환 시). 단 별도 작업(§6) |

요지: **A안은 "글 수 × 동시편집"이 커질수록 단일 index.json이 병목·사고 지점**이 된다.
B안은 그 지점이 구조적으로 없다. 확장을 요구사항으로 못박은 이상 B가 맞다.

## 5. 권고 — B안(하이브리드), 단 과설계 금지

확정 권고: **메타 = DynamoDB, 본문 = S3, 이미지 = S3+CloudFront.**

근거 3줄:
1. 비용·읽기성능은 무승부(공개 정적·본문 S3) → 차별점은 편집 안전성·확장성뿐.
2. 직원 다수 동시편집 + 발행/임시 상태 + 향후 작성자·감사·예약발행 → 전부 레코드 단위
   저장(B)에서 자연스럽고, 단일 파일(A)에서 부자연스럽다.
3. 마이그레이션 스크립트가 이미 거의 완성. 추가 비용 낮음.

### 확정 스키마 — `isuhani-journal-posts` (On-Demand)

```
PK  postId            string   기존 13자리 logNo 유지, 신규는 시간정렬 ID(KSUID/ULID)
─ 속성 ─
  title             string
  category          string
  status            "PUBLISHED" | "DRAFT" | "HIDDEN"   발행/임시저장/숨김(소프트삭제)
  date              string   ISO "2026-04-24"   정렬 키 소스(노출 날짜)
  addDate           string   "2026. 4. 24."     표시용
  thumbnail         string|null
  excerpt           string|null
  bodyKey           string   "posts/{postId}.json"   ← S3 본문 포인터(본문은 DDB에 없음)
  bodyKind          "cms" | "html"
  author            string|null    작성자(향후 직원 식별)
  images            list<string>   본문 내 이미지 키(옵션, 정리·교체용)
  createdAt / updatedAt   ISO
  updatedBy         string|null    마지막 수정자(감사)

GSI1  전체 최신순     GSI1PK="POST"            GSI1SK=date
GSI2  카테고리별      GSI2PK=category          GSI2SK=date
GSI3  상태별 관리     GSI3PK=status            GSI3SK=date   ← 직원 "임시저장함" 등
```

접근 패턴 매핑: Q1→GSI1, Q2→GSI2, Q3→GSI3, Q5→GetItem+S3, Q6→Put/UpdateItem(부분)+S3 put.
부분수정은 `UpdateItem`으로 바뀐 필드만(제목만, 카테고리만) 원자적 갱신 → 동시편집 안전.

**과설계 경계(지금 안 함):** 풀 single-table 디자인(여러 엔티티 한 테이블)은 콘텐츠 타입이
실제로 늘 때 도입. 지금은 글 한 종류라 위 단일 테이블로 충분. `status`에 `DRAFT` 자리만
미리 비워둬 임시저장 UX는 바로 받침. 예약발행은 필요해질 때 `publishAt` 속성 추가로 확장.

### 이미지 — 본문에 URL 박기 전에 결정 (Phase B 게이트)

지금 이관본은 직접 S3 URL(`isuhani-clinic-data.s3.ap-northeast-2.amazonaws.com/images/legacy/...`).
이대로 본문에 박으면 버킷명·리전에 결합되고 캐싱(CDN) 이점이 없다. 본문 6천여 곳에 한 번
박으면 되돌리기 번거로우므로 **Phase B 전에 이미지 URL 전략을 확정**한다.

| 옵션 | URL | 장단 |
|---|---|---|
| i. 직접 S3 | `...s3...amazonaws.com/images/...` | 즉시 가능 · 버킷결합·무캐시 |
| ii. 사이트 CloudFront 경유(권장) | `https://isuclinic.co.kr/cdn/images/...` | 동일출처·캐시·버킷은닉. CF behavior 1개 추가 |
| iii. 전용 이미지 도메인 | `https://img.isuclinic.co.kr/...` | 가장 깔끔 · DNS+CF 설정 |

권장 = ii. 기존 사이트 CloudFront에 `/cdn/*` → 데이터버킷 오리진 behavior 하나 붙이고
본문엔 `/cdn/images/legacy/...` 같은 **상대경로성 안정 URL**을 박는다. 나중에 버킷을 옮겨도
본문 불변.

## 6. 공개 사이트 읽기 경로 — 정적 유지 권고

B안이라고 공개를 꼭 동적으로 바꿀 필요 없다. 두 갈래:
- (가) **정적 재빌드 유지(권장, 80점)** — 발행 시 CodeBuild가 DynamoDB+S3로 글을 모아
  정적 export. 읽기비용 0·견고·현행 파이프라인 재사용. 단점: 발행→반영 2~5분 지연.
- (나) 동적 읽기 — 공개 페이지가 API로 DynamoDB 조회. 즉시 반영이지만 읽기경로 재작성 +
  상시 Lambda/읽기비용. 이 시스템엔 과함.

권장 (가). "즉시 반영"이 절실해지면 그때 (나) 또는 ISR류로 승급.
